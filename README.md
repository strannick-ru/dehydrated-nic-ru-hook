# Хук для DNS-валидации через NIC.RU

Хук для автоматического прохождения DNS-01 проверки при выпуске сертификатов Let's Encrypt.
Работает через API NIC.RU: создает нужные TXT-записи, ждет их появления в DNS и удаляет после проверки.

Из-за известных проблем с рассинхронизацией NS у NIC.RU (разные серверы имен могут временно отдавать разные данные) в хуке сделана двухуровневая проверка:

1. сначала хук опрашивает авторитативные NS зоны и ждет, пока хотя бы один из них покажет все нужные TXT-записи;
2. затем хук опрашивает внешние DNS (`1.1.1.1` и `8.8.8.8`) и ждет, пока каждый из них покажет все записи.

Только после этого управление передается обратно ACME-клиенту.

## Что нужно для работы

- аккаунт на [nic.ru](https://www.nic.ru/), у которого есть доступ к нужной DNS-зоне;
- OAuth-приложение, зарегистрированное в NIC.RU (об этом ниже);
- утилиты `curl`, `jq`, `xmllint` и `dig` в системе, где запускается хук.

## Как получить доступ к API NIC.RU

### 1. Убедитесь, что у вас есть аккаунт с нужной зоной

Зайдите в [личный кабинет NIC.RU](https://www.nic.ru/manager/) и проверьте, что нужная DNS-зона видна и доступна для редактирования в разделе DNS-хостинга.

Логин и пароль от этого аккаунта понадобятся хуку:

| Что                        | Переменная окружения |
|-----------------------------|----------------------|
| Логин аккаунта NIC.RU       | `NICRU_user`         |
| Пароль аккаунта NIC.RU      | `NICRU_pass`         |

Если зона находится в другом аккаунте или у вашего пользователя нет прав на DNS-хостинг, токен может выдаться, но запись создать не получится.

### 2. Зарегистрируйте OAuth-приложение

Откройте страницу OAuth-сервера NIC.RU:

<https://www.nic.ru/manager/oauth.cgi?step=oauth.app_list>

Подробное описание процедуры есть в справке NIC.RU:

<https://www.nic.ru/help/oauth-server_3642.html>

На этой странице нажмите кнопку создания нового приложения. При создании:

- укажите понятное название, например `acme-dns-hook`;
- скопируйте и сохраните идентификатор приложения и секретное слово сразу после создания — потом секретное слово уже не будет видно.

Полученные значения нужны хуку:

| Что                                   | Переменная окружения |
|----------------------------------------|----------------------|
| Идентификатор приложения (`application`) | `NICRU_id`           |
| Секретное слово приложения (`password`)  | `NICRU_secret`       |

### 3. Проверьте, что приложение имеет доступ к DNS

Хук работает с DNS-хостингом через адреса вида `/dns-master/...` и при получении токена запрашивает такие права:

```
(GET|PUT|POST|DELETE):/dns-master/.+
```

Обычно этого достаточно, если аккаунт имеет доступ к DNS-хостингу. Если у вас в NIC.RU настроены ограничения на доступ к API, убедитесь, что приложение может работать с DNS-записями.

## Настройка переменных окружения

Хуку нужно пять переменных:

```bash
export NICRU_url="https://api.nic.ru"
export NICRU_user="ваш-логин"
export NICRU_pass="ваш-пароль"
export NICRU_id="идентификатор-приложения"
export NICRU_secret="секретное-слово-приложения"
```

Если хук запускается из CI, лучше передавать секреты через защищенные переменные пайплайна, а не хранить их в файлах репозитория.

## Проверка доступа

Перед первым запуском хука полезно убедиться, что NIC.RU выдает токен:

```bash
curl --silent --show-error -X POST "${NICRU_url}/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "grant_type=password" \
  --data-urlencode "username=${NICRU_user}" \
  --data-urlencode "password=${NICRU_pass}" \
  --data-urlencode "scope=(GET|PUT|POST|DELETE):/dns-master/.+" \
  --data-urlencode "client_id=${NICRU_id}" \
  --data-urlencode "client_secret=${NICRU_secret}"
```

Если всё в порядке, в ответе будет JSON с полем `access_token`.

Чтобы заодно проверить доступ к DNS, можно запросить список зон:

```bash
TOKEN="$( \
  curl --silent -X POST "${NICRU_url}/oauth/token" \
    -H "Content-Type: application/x-www-form-urlencoded" \
    --data-urlencode "grant_type=password" \
    --data-urlencode "username=${NICRU_user}" \
    --data-urlencode "password=${NICRU_pass}" \
    --data-urlencode "scope=(GET|PUT|POST|DELETE):/dns-master/.+" \
    --data-urlencode "client_id=${NICRU_id}" \
    --data-urlencode "client_secret=${NICRU_secret}" \
  | jq -r '.access_token' \
)"

curl --silent -H "Authorization: Bearer ${TOKEN}" \
  "${NICRU_url}/dns-master/zones"
```

Если в ответе видны ваши зоны, значит доступ настроен правильно.

## Особенности проверки DNS-записей

У NIC.RU бывает так, что разные NS временно показывают разные наборы TXT-записей для одного и того же имени `_acme-challenge`. Из-за этого хук не заканчивает работу после первого успешного ответа от DNS.

Проверка устроена в два этапа:

1. **Авторитативные NS зоны** — хук определяет серверы имен зоны (через `NICRU_ns_resolver`, по умолчанию `77.88.8.8`) и по очереди опрашивает каждый. Как только хотя бы один из них отдает полный набор ожидаемых TXT-записей, первый этап считается пройденным. Если ни один не ответил за отведенное время, хук все равно переходит ко второму этапу.
2. **Внешние DNS** — хук опрашивает каждый из внешних DNS-серверов (`1.1.1.1` и `8.8.8.8` по умолчанию) и ждет, пока **каждый** из них покажет **все** ожидаемые TXT-записи. Пока хотя бы один сервер не видит полного набора, хук продолжает ждать.

Записей может быть несколько: например, если в сертификат входят и основной домен, и wildcard (`example.com` + `*.example.com`), оба используют одно и то же имя `_acme-challenge.example.com`, но с разными значениями TXT. Хук ждет, пока все значения станут видны.

Это нужно, потому что Let's Encrypt при проверке может попасть на любой публичный DNS, и если хотя бы один из них еще не видит все записи, проверка может не пройти.

При необходимости таймауты и список серверов можно переопределить:

| Переменная                    | По умолчанию       | Что делает                                                  |
|-------------------------------|---------------------|-------------------------------------------------------------|
| `NICRU_ns_resolver`           | `77.88.8.8`         | Через какой DNS узнавать авторитативные NS зоны             |
| `NICRU_authoritative_timeout` | `300` (5 мин)       | Сколько секунд ждать появления записи на NS NIC.RU          |
| `NICRU_public_timeout`        | `1800` (30 мин)     | Сколько секунд ждать появления записи на внешних DNS        |
| `NICRU_public_resolvers`      | `1.1.1.1 8.8.8.8`  | Список внешних DNS для второго этапа проверки (через пробел) |

## Частые ошибки

### `Failed to obtain access token`

Не удалось получить токен. Обычно причина в одном из:

- неверный логин или пароль аккаунта NIC.RU;
- неверный идентификатор или секретное слово приложения;
- приложение зарегистрировано в другом аккаунте.

### `Zone not found for domain ...`

Хук не нашел зону для указанного домена. Проверьте:

- видна ли зона в личном кабинете NIC.RU в разделе DNS-хостинга;
- совпадает ли название зоны с тем, что вы ожидаете;
- доступна ли зона именно из этого аккаунта.

Для диагностики запросите список зон тем же токеном (см. раздел «Проверка доступа» выше).

### Токен выдается, но запись не создается

Скорее всего, аккаунт не имеет прав на редактирование зоны в DNS-хостинге, хотя токен для API получить удалось.

### Один DNS видит запись, а другой — нет

Это нормальная промежуточная ситуация для NIC.RU. Именно поэтому на втором этапе хук ждет, пока каждый из внешних DNS покажет полный набор записей, а не останавливается после первого успешного ответа.
